# Enforce 100-line file limit across codebase

## Problem

Files exceeding 100 lines signal SRP violations — too much responsibility in one place. Long files are harder to review, test, and reason about. Catching this early prevents accumulation of god-files.

## Scope

Enforce on all source files:
- `web/src/**/*.{ts,tsx}` — TypeScript / React components
- `server/**/*.rs` — Rust backend

Exclude:
- Generated files (e.g. `*.generated.ts`, `*.pb.rs`)
- Test fixtures / snapshots
- Config files (e.g. `tailwind.config.ts`, `vite.config.ts`)
- `index.ts` barrel files (re-exports only, no logic)

## Implementation

### Web (TypeScript/TSX)
Add ESLint `max-lines` rule to `.eslintrc` / `eslint.config.*`:
```js
"max-lines": ["error", { "max": 100, "skipBlankLines": true, "skipComments": true }]
```

### Server (Rust)
Add CI step that fails on any `.rs` file exceeding 100 lines:
```bash
find server/src -name "*.rs" | xargs awk 'END{if(NR>100) exit 1}' 
```
Or use a `just` recipe / Makefile target.

### CI gate
Both checks must run in CI (lint + line-count script) so violations block merge.

### Pre-commit hook (optional but recommended)
Wire into `.husky` or `lefthook` so devs get immediate feedback.

## Acceptance Criteria

- [ ] ESLint `max-lines` rule enabled and failing on violations in web
- [ ] CI script or linter blocks Rust files > 100 lines
- [ ] All *existing* violations refactored into smaller files (split by responsibility)
- [ ] CI pipeline enforces the rule on every PR
- [ ] CLAUDE.md updated with the 100-line rule so future AI sessions know the constraint

## Notes

- 100 lines = logical lines (skip blanks + comments per ESLint config)
- When splitting files, prefer extracting by domain concept, not arbitrary line count — the limit is a signal, not a target
- Components with large JSX trees should extract sub-components; hooks with many effects should split into focused hooks
