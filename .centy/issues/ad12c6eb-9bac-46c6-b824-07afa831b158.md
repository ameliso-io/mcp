# Allow users to edit slugs (suites, cases, runs)

## Problem

All slugs are immutable after creation. As projects evolve, users are stuck with stale identifiers across suites, case paths, and runs.

---

## Entities in Scope

| Entity | Current Slug | Storage | Complexity |
|--------|-------------|---------|------------|
| Suite | `suites/{slug}.yaml` | PK in `suites(repo_id, slug)`, FK in `suite_cases` | Medium |
| Case | `cases/{path}.md` | PK in `cases(repo_id, case_path)`, FK in `suite_cases` + results | High (hierarchical) |
| Run | `runs/{YYYY-MM-DD}-{slug}/` | PK `run_id` in `runs`, all results nested under dir | High (historical record) |

---

## Suite Slug Rename

### Proto
```protobuf
message UpdateSuiteRequest {
  string repo_id   = 1;
  string slug      = 2;   // current slug (lookup key)
  // ...existing fields...
  optional string new_slug = 7;
}
```

### Repo layer (`repo.rs`)
1. Validate `new_slug` via `validate_slug_path()`
2. Check no existing suite with `new_slug` in same `repo_id` → conflict error
3. In a transaction:
   - `UPDATE suites SET slug = ? WHERE repo_id = ? AND slug = ?`
   - `suite_cases` uses `(repo_id, suite_slug)` FK — verify `ON UPDATE CASCADE`; if absent, update manually
   - Rename `suites/{old}.yaml` → `suites/{new}.yaml`

### Frontend (`SuitesTab.tsx`)
- Add `slug` input to edit form (lines 323–354), pre-populated with current slug
- Client-side validation: `/^[a-z0-9_-]+$/`
- Warning: "Renaming the slug will break CI scripts and API calls referencing the old slug."
- On success: replace `?expanded={old}` in URL with new slug

## Case Path Rename

### Complexity
Case paths are hierarchical (`auth/login`, `auth/signup`). A rename can be:
- **Leaf rename**: `auth/login` → `auth/signin` — single file
- **Prefix rename**: `auth` → `authentication` — renames all cases matching `auth/%`

### Proto
```protobuf
message UpdateCaseRequest {
  string repo_id    = 1;
  string case_path  = 2;   // current path (lookup key)
  // ...existing fields...
  optional string new_path = 8;
}
```

### Repo layer
1. Validate `new_path` via `validate_slug_path()` (allows `/`)
2. Check no existing case with `new_path` in same repo
3. In a transaction:
   - `UPDATE cases SET case_path = ? WHERE repo_id = ? AND case_path = ?`
   - Update `suite_cases`: `UPDATE suite_cases SET case_path = ? WHERE repo_id = ? AND case_path = ?`
   - Rename `cases/{old_path}.md` → `cases/{new_path}.md` (ensure parent dir exists)
4. **Prefix rename** (v2): if `rename_prefix = true`, apply `REPLACE(case_path, old_prefix, new_prefix)` to all matching rows + rename all files under directory

### Frontend
- Add `path` input to case edit form, pre-populated with current path
- Client-side validation: `/^[a-z0-9_\-]+(\/[a-z0-9_\-]+)*$/`
- Warning: "Renaming the path will break suite case lists and any direct references."

## Run Slug / Run ID Rename

> **Note:** Run slugs become part of the immutable `run_id` (`{YYYY-MM-DD}-{slug}`), which is the PK for `runs` and the filesystem directory. Renaming is allowed but carries risk: any external references (CI logs, links) pointing to the old `run_id` will break.

### Proto
```protobuf
message UpdateRunRequest {
  string repo_id  = 1;
  string run_id   = 2;   // current run_id (lookup key)
  optional string new_slug = 5;   // new slug portion only; date prefix is preserved
}
```

Server reconstructs `new_run_id = "{date_prefix}-{new_slug}"`, preserving the date.

### Repo layer
1. Parse date prefix from current `run_id`
2. Validate `new_slug` via `validate_slug_path()`
3. Construct `new_run_id = "{date}-{new_slug}"`
4. Check no existing run with `new_run_id` in same repo
5. In a transaction:
   - `UPDATE runs SET run_id = ? WHERE repo_id = ? AND run_id = ?`
   - Update `run_results` (FK cascade or manual)
   - Rename `runs/{old_run_id}/` → `runs/{new_run_id}/`

### Frontend (`RunsTab.tsx`)
- Add rename action to run row (kebab menu)
- Input: new slug only (date prefix shown as read-only prefix label)
- Strong warning: "Renaming a run changes its permanent ID. External links and CI references to the old ID will break."

---

## Shared Edge Cases

| Case | Handling |
|------|----------|
| New slug/path already exists | Conflict error from server, surface as form error |
| File rename fails after DB update | Rollback DB update in transaction |
| DB update fails after file rename | Rollback file rename |
| `new_slug` empty / unchanged | Treat as no-op, skip rename logic |
| Parent directory missing after rename | Create intermediate dirs before rename |

---

## Implementation Order

1. Suite slug rename (lowest risk — no cascading children)
2. Case path rename (leaf only first, prefix rename as v2)
3. Run ID rename (highest risk — rename historical record last)
